# Project: `ggmemo` v0.2 — Design improvements and new features

## Context

v0.1.0 shipped with two functions (`annotate_callout()`, `annotate_change()`)
that add callout labels and change annotations to ggplot2 business charts.
The API works, tests pass, the vignette and pkgdown site are deployed.

v0.2 addresses the design debt identified during code review, dogfooding,
and community feedback. The changes fall into three categories: removing
heuristics that guess user intent, making the API more consistent with
ggplot2 conventions, and adding the one most-requested feature.

**Read these files first:**
- `notes/DESIGN.md` (v0.2 ideas section at the bottom)
- `notes/code-walkthrough.md` (architectural context)
- `R/annotate-callout.R` and `R/annotate-change.R` (current implementation)

## The three design problems to solve

### 1. The data redundancy problem

Users pass `data` to both `ggplot()` and `annotate_*()`:
```r
ggplot(revenue, aes(x = quarter, y = revenue)) +
  geom_col() +
  annotate_change(revenue, from = ..., to = ..., value = revenue)
```

`revenue` appears twice. Every other ggplot2 layer inherits data from the
plot — our functions are the exception. This is the most common source of
confusion and the biggest barrier to feeling "native" in ggplot2.

**Options to explore:**
- Make `data` optional. If omitted, attempt to extract data from the plot
  object. This may require converting both functions into proper ggplot2
  `Stat`/`Geom` subclasses rather than standalone functions that return layers.
- Accept a plot object as the first argument: `annotate_change(p, from = ..., to = ...)`.
- Keep the current API but document the rationale clearly.

**Constraint:** Whatever approach you choose must still work when added with `+`.
The ggplot2 `+` operator is non-negotiable.

### 2. The x-column heuristic problem

`annotate_change()` guesses which column is the x-axis using
`detect_x_column()` — prefers Date > numeric/factor > character. This works
on simple data frames and fails on wide ones. `annotate_callout()` has the
same issue via `estimate_nudge()`.

**Options to explore:**
- Add an explicit `x` parameter: `annotate_change(data, x = quarter, from = ..., to = ..., value = revenue)`. Removes all guessing. Makes the API one argument wider but fully predictable.
- If solving problem #1 (data inheritance), extract the x mapping from the plot's `aes()` and use it directly. This is the cleanest solution but depends on solving #1 first.
- Accept `mapping = aes(x = quarter, y = revenue)` as in the original DESIGN.md spec. This was rejected in v0.1 in favor of `value`, but it would solve both the x and y column identification in one argument.

### 3. The `...` asymmetry

In `annotate_callout()`, `...` passes to the entire layer (`ggpp::geom_label_s()`).
In `annotate_change()`, `...` passes only to the label layer, not the segment.
A user who passes `linewidth = 1` expecting a thicker arrow gets nothing.

**Options to explore:**
- Split `...` into `label_args` and `segment_args` named list arguments.
- Pass `...` to both layers and let ggplot2 ignore irrelevant params (may produce warnings).
- Add specific named arguments for the most common segment overrides (`arrow_color` is already handled by `colors`, so maybe `arrow_linewidth` and `arrow_style`).

## Features to consider

### `annotate_milestone()` (most requested)

A vertical reference line with a label — "Product launch", "Policy change",
"COVID lockdown". Common in business time series. Simpler than the other
two functions: no data lookup needed, just an x position and a label.

```r
# Proposed API
ggplot(data, aes(x = date, y = revenue)) +
  geom_line() +
  annotate_milestone(x = as.Date("2024-03-15"), label = "Product launch")
```

### Colorblind mode

The current green/red defaults are not colorblind-safe. A `colorblind = TRUE`
argument (or a package-level option) that switches to blue/orange would be
straightforward. The `colors` argument already supports custom palettes, so
this is mostly about providing a good default alternative.

### `...` pass-through for the segment layer

See design problem #3 above.

## Things to NOT do in v0.2

- Don't add facet support. That's a v0.3 problem at earliest.
- Don't submit to CRAN yet. Get real user feedback on v0.2 first.
- Don't rewrite the formatting system. `format_number()`, `format_percent()`, etc. work fine. If they need to get more sophisticated, consider depending on the `scales` package rather than building a formatting framework.
- Don't add a theme. ggmemo is about annotations, not aesthetics.

## How to work

- Start by prototyping solutions to design problem #1 (data inheritance).
  This is the hardest and most impactful change. If you solve it, problems
  #2 and #3 may become simpler or unnecessary.
- Write tests first for the new behavior, then implement.
- Maintain backwards compatibility where possible. If the v0.1 API must
  break, document the migration path in NEWS.md.
- Run `devtools::check()` after every change. Zero errors, zero warnings.
